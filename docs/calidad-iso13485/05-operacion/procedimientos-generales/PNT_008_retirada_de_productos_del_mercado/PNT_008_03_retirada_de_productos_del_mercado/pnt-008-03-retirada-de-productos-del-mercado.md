# Objetivo

El objetivo del presente procedimiento es definir las actividades a
realizar para retirar un producto del mercado por parte de Spika Tech,
adelante SPIKA, de forma rápida y eficiente. Todo ello si existen
pruebas reales de que el producto pueden constituir un peligro para la
seguridad de los clientes.

# Alcance

El alcance del procedimiento es a todos los productos fabricados por
SPIKA.

# Responsabilidades y Departamentos afectados

-   La Dirección y el Técnico Responsable son los responsables de
    decidir si se llevan a cabo los procesos de retirada.

-   El Técnico Responsable es responsable de:



-   Comunicarse con las autoridades

-   Notificar a la parte de distribución el
    bloqueo/inmovilización/inactivación/ cancelación de acceso, etc...,
    según el caso, de los productos afectados

-   Hacer seguimiento y evolución del proceso de recuperación del
    mercado.

# Procedimiento

Todas las alertas de seguridad recibidas que puedan tener un impacto
perjudicial tras el uso de los productos de SPIKA deben ser evaluadas
por la Dirección y el Técnico Responsable. Adicionalmente, pueden
recibirse notificaciones de reclamaciones que puedan escalarse mediante
este procedimiento.

Si la decisión final da lugar a que el producto sea bloqueado, retirado
del mercado, inavilitado para su acceso, desactivado, etc..., según el
caso, se realizará como se indica en este procedimiento.

# Alcance del proceso de retirada

Una vez que se ha tomado la decisión de retirar del mercado y/o
inhabilitar un producto, según el caso, las acciones deben ser rápidas y
eficaces, a fin de evitar la propagación del riesgo para la salud y
protección de los clientes y reducir en la medida de lo posible las
consecuencias que puedan producirse.

Las retiradas pueden deberse a:

-   Una reclamación o defecto de calidad detectados dentro de la empresa

-   Una reclamación externa por un defecto de calidad (ver **PNT-007
    Gestión de Reclamaciones**)**

-   Requisito de las autoridades sanitarias correspondientes.

# Tipos de retirada

# Retirada urgente

Cuando existe una probabilidad razonable de que el uso del producto
pueda causar daños irreparables a la salud de las personas.

En este caso, el objetivo principal es recoger y/o inhabilitar TODOS los
productos que han salido de a uso, incluidos los que pueden haber sido
enviados al público, utilizando todas las medidas necesarias, incluida
la publicidad en los medios de comunicación si es necesario.

# Retirada no urgente

Una situación en la que el uso del producto puede causar efectos
adversos temporales para la salud que son reversibles y donde la
probabilidad de efectos irreversibles es remota.

En este caso el objetivo fundamental es el bloqueo y/o inhabilitación
inmediata de todos los elementos de sus destinos. Los medios de
comunicación no se utilizarán excepto en casos específicos de alto
riesgo.

# Aviso a las autoridades sanitarias

Una vez tomada la decisión de retirar y/o inhabilitar un producto o lote
del mercado, es obligatorio informar inmediatamente a las Autoridades
sanitarias.

La AEMPS u otras entidades sanitarias delegadas a SPIKA se notificarán
por correo electrónico y se utilizará la información proporcionada para
evaluar los posibles riesgos, siendo el Técnico Responsable el encargado
de hacer estas comunicaciones.

# Información necesaria para enviar el aviso

Después de la decisión de retirar un producto de productos del mercado,
será necesario recopilar una serie de información que se enviará a las
Autoridades sanitarias, que generalmente serán las siguientes:

-   Datos por lotes de producto (si es un producto físico)

-   Nº de lote de producto (si es un producto físico) o versión del
    mismo (si es un producto software)

-   Motivo de la decisión

-   Evaluación del riesgo

-   Datos de producción, que incluye:



-   Número de unidades totales producidas (si es un producto físico) o
    número de licencias vendidad (si es un producto software)

-   Lista de clientes suministrados

-   Nombre de los clientes y datos de contacto: teléfono, email,
    dirección postal, etc…

-   Fecha de distribución/activación de licencia de uso

-   Albaranes o documentación relacionada con las entregas/activaciones
    de licencia.

# Bloqueo de producto

Podrá realizar el bloqueo/inhabilitación total o parcial de productos
hasta resolución de la investigación abierta por la reclamación o
defecto notificado. El bloqueo/inhabilitación parcial debe estar
justificado o si ha sido posible acotar el defecto. También pudieran
realizarse bloqueos/inhabilitaciones preventivas que se levantarán en
caso de descartar el defecto. A continuación se detalla el proceso de
bloqueo/inahibilitación del software que se realizará en remoto en caso
de necesidad:

-   Paso 1: sabiendo la versión del software a bloquear/inhabilitar, el
    personal de IT de Spika accederá a la base de datos de la
    aplicación, donde se almacenan todas las versiones disponibles del
    software.

-   Paso 2: una vez en la base de datos y habiendo identificado la
    versión a bloquear/inhabilitar, se procederá a su deshabilitación
    cambiando los parámetros de acceso que se consultan cada vez que se
    inicia el software.

-   Paso 3: tras modificar estos parámetros, la versión del software
    quedará bloqueada/inhabilitada, impidiendo al usuario acceder a
    dicha versión.

-   Paso 4: una vez bloqueada/inhabilitada, el personal de Spika
    comprobará que el usuario no puede utilizar la versión del software
    mediante la ejecución de pruebas de acceso, dando así por concluido
    el proceso de bloqueo remoto del software.

# Informe de retirada/inhabilitación y eficacia de la misma

Junto con la documentación generada por la reclamación se debe generar
un informe donde se recoja toda la trazabilidad de las unidades
recuperadas/inhabilitadas.

En este informe se debe incluir como mínimo:

-   Número de unidades distribuidas (si es un producto físico) o
    versiones afectadas (si es un producto software)

-   Número de unidades afectadas (si es un producto físico) o versión
    afectada (si es un producto software)

-   Número de unidades recuperadas/versiones inhabilitadas, en caso de
    discrepancia con las cantidades totales debe explicarse y evaluar el
    riesgo

-   Riesgo final de la retirada/inhabilitación

-   Clientes totales

-   Clientes contactados

-   Nº de lote/versión

-   Producto afectado

En el informe se debe incluir una conclusión de si la
retirada/inhabilitación se ha considerado eficaz o no para ello además
de la reconciliación final de unidades/versiones se debe evaluar si se
ha podido contactar con todos los clientes para ser informados y que se
hayan llevado a cabo las medidas adecuadas para proceder con la
retirada/inhabilitación.

# Simulacro

Si no se ha producido ninguna retirada/inhabilitación en el transcurso
de un año natural, se realizará un simulacro por el personal de SPIKA,
con el fin de comprobar la eficacia de la implantación de este
procedimiento y los métodos de comunicación con los clientes.
